Introduction

Gorilla IO is a proven game developers software architecture. with over 25 years of expertise in building over 45 published game titles. The architecture addresses fundamental software mechanics, structures and relationships needed to efficiently build games, game engines and multimedia software systems. Concepts apply to any modern programming language and multimedia platforms from the lowest firmware level to a higher utility level provided by professional game engines such as Unreal and Unity (ideal for Interop).

A primary objective of Gorilla IO architecture is to partition software into solid modules called services that service the application cycle, also known as the game loop. Services are engineered to facilitate building application content for optimal IO (input/output) performance in a coequal OOD environment as code blocks easily pipelined by agile methodologies.

Overview

Gorilla IO defines a software architecture for universally building interactive media and game applications on any given computing platform. Implementations range from low level platforms (game console, smart devices) coded in hardware assembly and C languages. To mid-level platforms: PC, Mac and Unix(ITV) coded in C/C++ and Java languages. To high-level platforms such as a Web browser application coded in JavaScript. Mechanics

Mechanics describes structured data movement in segments of time, namely a frame. Each frame breaks the linear execution into small software parts to maintain the illusion that all frame computations happen at the same time or in the moment. Therefore, each software part assumes the same time when frame execution starts. This is called the Stage time which is stamped by the Stage before a frame cycle begins. Platforms can have a frame rate from 10 fps(frame per second) minimum needed to maintain the illusion of motion on low end systems to over 120 fps on advanced platforms. Typical systems have a target of 60 fps of 16ms (milliseconds) which we will use, going forward, as the example. See the diagram of a frame illustrated by a gear like clock with divided into 16 for each millisecond. Each section slice, called a Service, is processed in sequence until the last section to complete the frame cycle. Then a new frame begins and so forth. Services are unique black boxes executed in sequence. It is said that when a service is processing, it has the focus. The arrow moves in a clockwise direction initially pointing to a the slice of operation data for the Stage Service in focus for the first 2ms (1-2) frame section. After that the Message Service will take focus to process 4ms (3-6). Then respectfully Task Service 2ms (7-8), Render Service 2ms(9-10) and Scene Service 2ms (11-12) The remaining 4ms section grayed out is not a service therefore no service focus or processing until the frame starts again at millisecond (1).

Design

Software is designed by partitioning content and engine code. Engine code is further partitioned into black box components call services controlled by the Stage platfrom manager. The Stage is a special service that layers the platform as the services controller. Actors are content code aggregates of service supplied content objects called agents. Services are considered engine modules. Content coding is the act of scripting agents. Content code communicates with services indirectly via agents that conform to the architecture principles. The Agent is an Actor content object aggregate registered to a service that manipulates state each frame cycle.

For the example below: The Scene service has transitioned to the Game Scene. The Game Scene constructs by subscribing to the Task service for content logic scripting and aggregating actors Background Actor and Player Actor. The Background Actor subscribes to the Render service to display image. The Player Actor subscribes to the Input service for user control, Task service for scripting and Render service to display image animations.

Principles

Gorilla IO architecture roots at services. Principles adhere to OOP trending to a data-driven approach as opposed to event-driven architectures.

  • Moment Principle
        A moment in time is measured by the amount of time it takes to process a frame, typically 16ms. Services execute code linearly by the Stage time constant stamped before the frame begins to process. This maintains the illusion of a moment in time, as all content logic is based on Stage time. For finer time granularity (less than 16ms), a service can generate sequenced time data for content logic consumption.  
  • Aggregation
         Aggregation prefers composition over inheritance. Content objects are modules composited by service content objects and/or other content objects. For lower level programming languages such as (C/C++ or Objective C), the simple act of object aggregation will naturally optimize memory allocation. Aggregation simplifies object definitions. A benefit to the refactoring effort with intuitive re-composition.
    For example: extracting out code can be a matter of moving aggregated modules and logic to the new module. Game Scene keeps Scene Task then aggregates Background Actor and Player Actor.
  • Monotonic State
         Service Agent state, transformed at the service layer ready to use by content, remains unchanged (monotonic) during the frame cycle. Requests that impose on a service's internal state while that service has focus is posted to next cycle. Below example shows the Mouse service translates the Mouse content object data to position values X=10 and Y=40. The Mouse position values X=10 and Y=40 state are monotonic across Game Scene service content objects until the next Mouse service focus to update new values of X and Y.
  • Atomic Cohesion
         A Service has nearly perfect cohesion (atomic) All service state is internal, only the exposed operation method process() is called exclusively by the stage each cycle. Content communicates with a service by delegated aggregate service agents. Agents have degrees of functional cohesion depending on the class shape needed for content consumption. Scenes and Actors have procedural cohesion as they script agents which have functional cohesion to the service.
  • Synergic Multitasking
         Synergic multitasking, similar to cooperative multitasking with coroutines, partitions the loop process into logical cooperative pieces. Content object scripted routines are constructed by stringing these pieces separated by time into a linear procedure. Content code logic has the ability to yield to another task or sleep its own execution in milliseconds. Below illustrates a content Actor aggregated with a Task agent that subscribes to the Task service. Task logic yields to sleep for 0-n ms. Yielding at zero milliseconds guarantees wakeup at next cycle. Task service wakes up Task which then continues with content logic.
